## 1. Timer 模块重写

> 🔄 **重写** · 影响文件：`Assets/TEngine/Runtime/Module/TimerModule/`

### 问题

原定时器模块基于**线性遍历**查找到期定时器，定时器数量增长后效率低下，多定时器场景下形成性能瓶颈。频繁创建销毁 Timer 对象产生 GC 压力，且缺少运行时调试手段。

### 方案

| 改动 | 说明 |
|------|------|
| 最小堆调度 | 使用 `IndexedMinHeap` 替代线性遍历，定时器到期查找从 O(n) 降至 O(log n) |
| 对象池化 | `TimerNode` 对象池复用，消除频繁 new 产生的 GC |
| 诊断窗口 | Editor 窗口实时展示 Active / Zombie / Pool 定时器状态 |

### 状态

✅ 已完成 · 23 个 EditMode 测试覆盖

## 2. UI 数据绑定系统

> 🆕 **新增** · 影响目录：`Assets/GameScripts/HotFix/GameLogic/Module/DataBinding/`、`UI/LoginUI/`

### 问题

原 UI 刷新完全依赖手动调用 `OnRefresh()`，存在以下痛点：

- **数据变了 UI 不知道** — 玩家升级了，主界面等级没刷新
- **同一数据多处展示** — 金币在主界面、商店、背包都有显示，改一处要改三处
- **OnRefresh 粒度太粗** — 道具数量变了要刷新整个列表
- **手动调 OnRefresh 容易遗漏** — 业务逻辑改了数据忘记通知 UI

### 方案

引入轻量级响应式数据绑定模块，与现有 `OnRefresh` / `GameEvent` 共存：

| 组件 | 说明 |
|------|------|
| `BindableProperty<T>` | 响应式属性包装器，赋值自动通知订阅者 |
| `ObservableList<T>` | 响应式集合，增删改事件驱动 |
| `BatchScheduler` | 帧级批次合并，同帧多次赋值合并为一次回调 |
| `DataContext` | 多源聚合 + 格式转换，View 不直接依赖 Model |
| 便捷方法 | `BindText` / `BindInteractable` / `BindToggle` / `BindSlider` / `BindSprite` |
| MVE 四层架构 | Model → Service → DataContext → View，按业务单位组织文件 |

```
数据流：Model 赋值 → BatchScheduler 帧末合并 → DataContext 转换 → View 自动刷新
```

### 状态

✅ 已完成 · 102 个测试覆盖 · [详细文档](../UnityProject/repowiki/zh/content/UI系统/UI数据绑定.md)

## 3. 框架质量加固

> 🔧 **修复** · 影响模块：事件系统、资源模块、内存池

### 问题

| 模块 | 问题 |
|------|------|
| 事件系统 | 单个 handler 异常导致后续 handler 全部中断 |
| 资源模块 | `async void` 泄漏、UnloadAsset 空池无告警 |
| 内存池 | 线程安全隐患、异常类型不一致 |
| 通用 | 集合遍历中修改导致异常、命名不统一 |

### 方案

| 改动 | 说明 |
|------|------|
| EventDelegateData 异常隔离 | 内联 try-catch，单个 handler 异常不中断链 |
| ResourceModule 清理 | 移除死代码、async void → UniTaskVoid、增加空池告警 |
| MemoryPool 线程安全 | 统一 `GameFrameworkException`、计数器线程安全 |
| 代码规范 | 集合安全遍历、Singleton Key 改进、冗余 using 清理 |

### 状态

✅ 已完成 · 12 个测试覆盖
